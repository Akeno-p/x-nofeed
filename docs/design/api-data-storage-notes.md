# APIレスポンスのDB保存方針 思考メモ

## 結論

- X APIのレスポンスに含まれる全フィールドに対応するカラムを作るのではなく、必要最低限のカラムだけで構成する。
- いいね数やインプレッション数は保存(表示)しない。

## 検討した内容(全フィールドの保存)

### 全フィールド保存のメリット

- 後から「このフィールドも欲しかった」となった時に、APIを叩き直す必要がない（従量課金なので再取得にはコストがかかる）
- APIは1回叩けば全フィールド返ってくるので、保存するカラム数で料金は変わらない

### 全フィールド保存のデメリット

- **ネストされたデータの問題**: X APIはネストした形式でデータを返すことがある。使わないデータのために、それを正規化してテーブルを作り外部参照させるのは無意味
- **APIの仕様変更が頻繁**: X APIはレスポンスの形式が変わることが多い。過去に保存したデータと新しく取得したデータの構造が合わなくなるため、「後で使うかもしれないから取っておく」という前提が成り立ちにくい

### 問題にならなかった点

- DBへのアクセス量・ストレージ容量: 個人利用（ユーザー1人）なのでデータ量は微小。パフォーマンスの観点では全フィールド保存でも問題ない

## いいね数やインプレッション数の表示をするか？

X APIのポスト取得レスポンスには `public_metrics` というフィールドが含まれており、いいね数・リポスト数・表示回数などが返ってくるので保存・表示することは可能ですが、コンセプトとのぶれとコスト面での問題があると思いました。

- **コスト**: いいね数やインプレッション数は都度変化するため、最新の数値を確認するには、その都度APIを叩き直す必要があり従量課金のコストがかかります。
- **コンセプト**: このアプリの目的は「SNSからの刺激排除」なので、承認欲求を刺激する数値の表示を無くすことは、このアプリが解決したい問題そのものに近いと思いました。

```json
// public_metricsのフィールド
{
  "id": "1234567890123456789",
  "text": "承認欲求からの解放",
  "created_at": "2025-01-15T10:30:00.000Z",
  "public_metrics": {
    "retweet_count": 12,
    "reply_count": 5,
    "like_count": 87,
    "quote_count": 3,
    "bookmark_count": 9,
    "impression_count": 1240
  }
}
```
